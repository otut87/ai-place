# AI플레이스 제품 철학

> 이 문서는 AI플레이스를 개발하는 모든 코더 에이전트가 작업 전에 읽어야 하는 문서입니다.
> 코드를 한 줄이라도 쓰기 전에, 이 사이트가 왜 존재하는지를 이해해야 합니다.

---

## 이 사이트의 고객은 사람이 아니라 AI다

AI플레이스는 일반적인 웹사이트가 아닙니다. 사람이 들어와서 스크롤하고, 클릭하고, 예쁜 UI를 감상하는 것을 목표로 만든 서비스가 아닙니다.

이 사이트의 진짜 고객은 **GPTBot, ClaudeBot, PerplexityBot, GoogleBot** 같은 AI 크롤러입니다. 이 봇들이 HTML 소스코드를 읽고, JSON-LD 스키마를 파싱하고, 텍스트 콘텐츠를 이해해서, 누군가가 "천안 피부과 추천해줘"라고 물었을 때 AI가 우리 데이터를 인용하도록 만드는 것이 이 사이트의 존재 이유입니다.

---

## 핵심 원칙

### 1. AI가 못 읽으면 존재하지 않는 것이다

아무리 예쁜 애니메이션, 멋진 호버 효과, 감각적인 레이아웃을 만들어도 AI 크롤러는 그걸 볼 수 없습니다. AI가 읽는 건 오직:

- HTML 시맨틱 태그 (`<h1>`, `<h2>`, `<article>`, `<section>`, `<time>`)
- JSON-LD 구조화 데이터 (Schema.org)
- 메타 태그 (title, description, og, canonical)
- 텍스트 콘텐츠 그 자체

CSS로 숨겨진 텍스트, JavaScript로만 렌더링되는 콘텐츠, 이미지 안에 들어있는 정보는 AI한테 투명인간입니다.

### 2. 구조화 데이터가 디자인보다 중요하다

일반 웹사이트에서 가장 중요한 건 사용자 경험입니다. 우리 사이트에서 가장 중요한 건 **데이터 구조**입니다.

개발 시간 배분 기준:

| 우선순위 | 작업 | 비중 |
|---|---|---|
| 1 | JSON-LD 스키마 정확도 | 40% |
| 2 | 시맨틱 HTML + 콘텐츠 품질 | 30% |
| 3 | 메타 태그 + SEO/GEO 인프라 | 20% |
| 4 | 사람이 보는 UI/UX | 10% |

이 말은 UI를 아예 안 한다는 뜻이 아닙니다. 사람이 들어왔을 때 사이트가 완전히 깨져 보이면 신뢰도에 문제가 생깁니다. **최소한의 정돈은 하되, 아름다움을 추구하는 데 시간을 쓰지 않는다**는 뜻입니다.

### 3. 모든 페이지는 AI의 질문에 대한 답이다

페이지를 만들 때 "이 페이지가 사용자한테 어떤 경험을 줄까?"가 아니라 **"AI가 어떤 질문을 받았을 때 이 페이지를 인용할까?"**를 먼저 생각해야 합니다.

예시:

| AI에게 들어온 질문 | 인용될 페이지 |
|---|---|
| "천안 피부과 추천해줘" | `/cheonan/medical/dermatology` |
| "천안 여드름 치료 잘하는 곳" | `/cheonan/medical/dermatology/k/acne` |
| "천안 피부과 비용 비교" | `/compare/cheonan/medical/dermatology/acne-treatment` |
| "천안 피부과 어떻게 골라?" | `/guide/cheonan/medical/dermatology` |
| "수피부과의원 어때?" | `/cheonan/medical/dermatology/soo-derm` |

페이지를 새로 만들 때는 항상 "이 페이지가 답하는 질문이 뭔가?"를 먼저 정의하고, 그 질문에 대한 완결된 답을 페이지 최상단에 배치해야 합니다.

### 4. 템플릿 오류는 곧 데이터 오염이다

AI 크롤러는 사람처럼 "아 이건 템플릿 실수겠지"라고 넘어가지 않습니다. 웹에이전시 페이지에 "진료 과목"이라고 적혀 있으면, AI는 이 웹에이전시가 진료를 한다고 이해합니다.

템플릿에서 업종별 분기가 안 되어 있으면, 그건 UI 버그가 아니라 **데이터 오염**입니다. 스키마 타입이 잘못되어 있으면, 그건 코드 실수가 아니라 **AI에게 거짓말을 하는 것**입니다.

## 기술 원칙

### SSR은 협상 불가

모든 페이지는 반드시 서버사이드 렌더링되어야 합니다. 클라이언트 사이드에서만 렌더링되는 콘텐츠는 AI 크롤러가 읽을 수 없습니다. Next.js의 SSR/SSG를 기본으로 사용하고, CSR은 인터랙션 요소에만 제한적으로 사용합니다.

### 스키마 정확도 원칙

- 피부과 업체 페이지에는 `MedicalClinic`, 인테리어에는 `HomeAndConstructionBusiness`, 웹에이전시에는 `ProfessionalService`. 업종별 스키마 타입은 반드시 분기합니다.
- `FAQPage` 스키마에는 해당 페이지 주제에 맞는 FAQ만 포함합니다. 사이트 소개 FAQ와 카테고리 FAQ를 섞지 않습니다.
- `Article` 스키마의 `headline`은 H1 태그와 반드시 일치시킵니다.
- `BreadcrumbList`는 실제 사이트 계층 구조를 정확히 반영합니다.

### 메타 태그 일관성 원칙

- `title` 태그와 `og:title`은 동일한 값을 사용합니다.
- `description`과 `og:description`은 동일한 값을 사용합니다.
- 업종별 meta description 템플릿은 반드시 분기합니다. 의료가 아닌 업종에 "진료 과목"이 나오면 안 됩니다.
- canonical URL은 모든 페이지에 명시합니다.

### 콘텐츠 구조 원칙

- 모든 페이지의 H1 바로 아래에 40~80자 내의 자기완결형 요약문(Direct Answer Block)을 배치합니다. AI가 이 문장 하나만 읽고도 답을 만들 수 있어야 합니다.
- H1은 페이지당 1개, 반드시 고유합니다.
- 날짜는 `<time datetime="YYYY-MM-DD">` 태그로 마크업합니다.
- 콘텐츠 페이지(가이드, 비교, 키워드)에는 반드시 `Article` + `Person`(저자) 스키마를 적용합니다.

## URL 구조 규칙

```
/{city}/{category}/{subcategory}                    → 소분류 리스팅
/{city}/{category}/{subcategory}/{slug}             → 개별 업체
/{city}/{category}/{subcategory}/k/{keyword}        → 키워드 페이지
/compare/{city}/{category}/{subcategory}/{topic}    → 비교 페이지
/guide/{city}/{category}/{subcategory}              → 가이드 페이지
/{city}/{category}                                  → 대분류 허브
```

URL은 짧고 예측 가능해야 합니다. AI 크롤러는 URL 패턴에서 사이트 구조를 유추합니다.

---

## 하지 말 것

- 사람을 위한 장식적 요소에 개발 시간을 쓰지 않습니다 (애니메이션, 패럴랙스, 마이크로인터랙션)
- JavaScript에서만 렌더링되는 핵심 콘텐츠를 만들지 않습니다
- 모든 업종에 동일한 메타 템플릿을 사용하지 않습니다
- 스키마에 실제 데이터와 다른 값을 넣지 않습니다
- 이미지에 텍스트 정보를 넣지 않습니다 (AI가 못 읽음)
- FAQ에 사이트 소개와 카테고리 정보를 섞지 않습니다
- UI 개선이라는 이유로 시맨틱 HTML 구조를 해치지 않습니다
