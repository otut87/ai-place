AI Place Admin 전체 설계안
자동화 파이프라인(업체 등록)과 블로그 자동발행까지 포함한 운영자(혼자 운영) 관점에서 정리합니다. 33,000원/월 SaaS + AI 노출 최적화라는 사업 모델을 기준으로, "운영자가 하루 30분 안에 모든 의사결정을 끝낼 수 있는 어드민"을 목표로 잡았습니다. 결제는 신용카드 정기결제(빌링키) 단일 채널을 전제로 합니다.

1. Admin의 역할 재정의
현재 어드민은 "업체 CRUD 도구"에 머물러 있습니다. 자동화 파이프라인과 블로그 자동발행이 들어오는 순간, 어드민은 다음 4가지 축으로 재정의되어야 합니다.
첫째, 검수자(Reviewer) 콘솔입니다. AI가 생성한 업체 정보·블로그 글의 승인/반려 흐름이 핵심이 됩니다. 둘째, 모니터링 대시보드입니다. AI 크롤러 유입, 색인 상태, 결제·구독 상태를 한눈에 봐야 합니다. 셋째, 콘텐츠 운영실입니다. 업체·서비스·블로그·FAQ·태그를 일관된 UI로 다룹니다. 넷째, 고객·결제 관리(CRM 라이트) 입니다. 33,000원 정기결제 고객의 라이프사이클(등록→승인→갱신→이탈)을 추적하고, 결제 실패·카드 만료로 인한 이탈을 선제 방어합니다.

2. 정보 구조 (IA)
좌측 사이드바 네비게이션 기준으로 다음 구조를 권장합니다.
대시보드 (Overview)
검수 큐 (Review Queue)
 ├─ 업체 검수
 ├─ 블로그 검수
 └─ 서비스/FAQ 검수
업체 (Places)
 ├─ 전체 목록
 ├─ 카테고리/지역 트리
 └─ 신규 등록
블로그 (Blog)
 ├─ 발행 캘린더
 ├─ 초안/예약/발행
 └─ 토픽 큐
자동화 (Pipelines)
 ├─ 수집 작업
 ├─ 생성 작업 (LLM)
 ├─ 발행 스케줄
 └─ 실패/재시도
SEO & AI 노출
 ├─ Sitemap 상태
 ├─ AI 크롤러 로그 (GPTBot, ClaudeBot, PerplexityBot 등)
 └─ 색인 모니터링
고객 (Customers)
 ├─ 전체 고객
 ├─ 구독 상태별 보기
 └─ 문의/CS
결제 (Billing)
 ├─ 진행 중인 구독
 ├─ 결제 실패 / 재시도 대기
 ├─ 만료임박 카드 (30일 이내)
 ├─ 해지 예정
 └─ 결제 이력
설정 (Settings)
 ├─ 결제 정책 문구 관리
 ├─ 카테고리/지역 마스터
 ├─ 프롬프트 템플릿
 ├─ 디자인 토큰
 └─ 팀/권한
이 IA의 핵심은 두 가지입니다. "검수 큐"를 최상단에 두는 것(자동화가 돌면 운영자의 본업은 "만들기"가 아니라 "검수하기"로 바뀌기 때문)과, "고객"과 "결제"를 분리하는 것(고객은 사람·관계 중심, 결제는 빌링 운영 중심으로 시야가 다르기 때문)입니다.

3. 페이지별 필수 요소
3.1 대시보드 (/admin)
지금처럼 단순 런처가 아니라, "오늘 무엇을 해야 하는가"를 보여주는 화면이어야 합니다. 상단에는 검수 대기 수(업체 N건 / 블로그 N건 / FAQ N건), 오늘 발행 예정, 실패한 자동화 작업, 결제 실패 / 카드 만료임박 4개의 액션 카드. 중단에는 AI 크롤러 방문 추이(7/30일), 신규 색인된 페이지 수, 활성 구독자 수 / MRR / 이번 달 결제 실패 건수 / 30일 내 카드 만료 예정. 하단에는 최근 활동 로그(누가 / 무엇을 / 언제 — 사람 vs 자동화 구분).
3.2 검수 큐 (/admin/review)
이 화면이 사실상 가장 자주 쓰는 페이지가 됩니다. 좌측에 큐 리스트(생성일시·타입·대상 업체), 우측에 좌우 분할 뷰: 왼쪽엔 AI 원본, 오른쪽엔 편집 가능한 최종본. 단축키로 승인(⌘↵) / 반려(⌘⌫) / 다음(J·K) 처리. 반려 시 사유 태그(사실오류 / 톤매너 / SEO부적합 / 중복)를 선택해두면, 나중에 프롬프트 개선 데이터가 됩니다.
3.3 업체 목록 (/admin/places)
현재의 카드형보다 테이블형이 훨씬 효율적입니다. 컬럼은 상태(검수중 / 발행 / 비활성), 업체명, 지역 / 카테고리, 최근 수정일, 구독상태(무료 / 유료), AI 노출 점수(서비스 / FAQ / 태그 충족도 %), 액션. 상단에 다중 필터(지역×카테고리×상태×구독), 다중선택 후 일괄 비활성 / 태그추가 / 재생성 가능. 현재의 "활성 / 비활성 뱃지 두 개 동시 표시" 버그는 단일 상태 enum으로 정리.
3.4 업체 상세 / 편집 (/admin/places/[id])
탭 구조 권장: 개요 / 서비스 / FAQ / 태그 / 블로그 / SEO / 자동화 이력 / 변경 로그. 편집 시 자동저장 + 토스트 + 변경 diff 보기. 우측 사이드바에 "AI 노출 체크리스트"(이름·설명·전화·카카오·네이버·서비스 3개 이상·FAQ 5개 이상·태그 8개 이상·이미지) 진척도 바.
3.5 신규 등록 (/admin/register)
지금의 "도시 → 카테고리 → 소분류 → 업체명 검색" 플로우는 유지하되, 검색 결과에서 "AI 자동완성 등록" 버튼 추가. 이 버튼이 자동화 파이프라인의 트리거가 됩니다(수집 → 생성 → 검수 큐로 이동).
3.6 블로그 운영 (/admin/blog)
캘린더 뷰를 메인으로(월 / 주). 각 셀에 발행 예정 글의 제목·상태·대상 카테고리. 좌측에 토픽 큐(자동 생성된 주제 후보 + 수동 추가), 우측에 발행 통계. 글 편집은 마크다운 + 프리뷰 분할, 상단에 타깃 키워드 / 카테고리 / 내부링크 후보(자동 추천).
3.7 자동화 파이프라인 (/admin/pipelines)
운영자가 가장 불안해하는 영역이므로 투명성이 핵심입니다. 각 작업의 단계(수집 → 정제 → 생성 → 검증 → 대기)를 간트 / 플로우 형태로 시각화. 실패한 작업은 에러 메시지 + 입력 페이로드 + 재시도 버튼. 수집 소스(네이버 플레이스 / 카카오맵 / 구글 플레이스 / 공공API)별 API 쿼터 사용량과 일일 비용(LLM 토큰 사용량) 표시.
3.8 SEO & AI 노출 (/admin/seo)
robots.txt에 GPTBot, ClaudeBot, PerplexityBot, CCBot 등을 허용해둔 만큼, 봇별 방문 로그·hit URL·404 비율을 시각화. Sitemap 갱신 시각, 신규 / 제거 URL diff, JSON-LD 검증 통과율. "AI 답변 노출 추정"(특정 쿼리에서 Perplexity / Claude / ChatGPT가 우리 페이지를 인용했는지 수동 체크 + 기록) 섹션.
3.9 고객 관리 (/admin/customers)
업체와 1:N으로 묶인 결제 주체. 컬럼: 사업자명 / 연락처, 구독 상태(체험 / 유료 / 연체 / 해지), 빌링키 상태(정상 / 만료임박 / 무효), 다음 결제일, 최근 결제 결과(성공 / 실패 N회), LTV, 마지막 접점. 상세에서 빌링키 정보(카드사·뒷4자리·만료월), 결제 이력, 결제 실패 재시도 로그, 해지 사유, 고객 메모. CS 문의는 별도 인박스(이메일 / 전화 / 카톡 채널 통합은 후순위).
3.10 결제 (/admin/billing)
MRR 이탈 방어 전용 섹션입니다. 메인은 /admin/billing/failures — 결제 실패 큐. 컬럼: 고객명, 실패 사유(한도초과·유효기간만료·카드분실 등 PG 응답코드 매핑), 재시도 일정, 마지막 시도 시각, 액션(재시도 / 카드 재등록 링크 발송 / 해지 처리). 별도 화면으로 만료임박 카드(30일 이내) 리스트와 자동 안내 메일 발송 상태, 해지 예정 리스트(이번 달 / 다음 달), 결제 이력 전체 검색. PG 연동은 포트원 + 토스페이먼츠 조합을 기본 권장.
3.11 설정 (/admin/settings)
프롬프트 템플릿 관리가 의외로 중요합니다. 카테고리별(피부과 / 인테리어 / 정비소…) 프롬프트를 버전 관리하고, 변경 시 A/B 비교로 어느 프롬프트가 검수 통과율이 높은지 추적. 디자인 토큰 JSON 업로드 / 다운로드, 카테고리·지역 마스터 CRUD, API 키 관리.
결제 정책 문구 관리에서 다음 표준 문구를 단일 소스로 관리합니다:

"본 서비스 결제는 신용카드 정기결제로만 이루어지며, 카드매출전표가 부가가치세법상 적격증빙입니다. 별도의 세금계산서 및 현금영수증은 발급되지 않습니다."

이 문구는 결제 화면, 이용약관, 영수증 이메일, FAQ 4곳에 동일하게 노출되도록 어드민에서 한 번만 수정하면 모든 노출처에 반영되도록 구성합니다.

4. 공통 레이아웃·UX 원칙
좌측 고정 사이드바(아이콘 + 라벨, 접기 가능), 상단 글로벌 바에 검색(⌘K 커맨드 팔레트)·검수 큐 카운트 뱃지·결제 실패 카운트 뱃지·알림·프로필. 본문은 max-width 없이 풀폭 사용(어드민은 정보 밀도 우선). 모든 리스트는 테이블 + 다중필터 + 다중선택 + 일괄작업 + URL 쿼리 동기화(공유·북마크 가능).
피드백은 토스트(저장 / 실패) + 인라인 검증 + 변경 diff 3종으로 통일. 파괴적 액션(삭제 / 대량 비활성 / 강제 해지)은 이름 입력 확인 모달. 모든 변경은 활동 로그(audit log) 에 자동 기록(자동화 vs 사람 구분).
디자인은 공개 페이지의 Place Green을 액션 컬러로만 차용하고, 어드민 자체는 회색 베이스 + 흰 카드로 정보 가독성 우선. shadcn CSS 변수에서 --primary만 #00a67c로 오버라이드하면 일관성 확보됩니다.

5. 자동화 파이프라인이 어드민에 끼치는 영향
자동화가 들어오면 데이터 출처(source)와 신뢰도(confidence) 메타가 모든 필드에 따라붙어야 합니다. 예: 업체 설명 옆에 "AI생성·신뢰도 0.82·네이버플레이스 출처" 칩. 이 메타가 있어야 검수자가 어디를 우선 보는지 결정할 수 있습니다.
또한 재생성(Regenerate) 버튼이 거의 모든 텍스트 필드에 필요합니다. 옵션으로 톤(친근 / 전문 / 간결), 길이, 강조 키워드 지정. 블로그도 마찬가지로 개요만 재생성 / 본문만 재생성 / 전체 재생성 분리.
마지막으로 "안전장치": 자동발행은 기본 OFF, 카테고리·고객등급별로 켜고, 자동발행 전 24시간 검수 유예를 두는 옵션. 이게 있어야 혼자 운영하면서도 사고를 막습니다.
